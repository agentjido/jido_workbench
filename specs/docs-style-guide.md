# Jido Docs - Style Guide

Standards for all pages under `priv/pages/docs/`. Follow these for consistency across every guide, concept page, and reference doc.

---

## Page structure

Every page follows this skeleton:

```
Frontmatter (Elixir map)
---

[Opening paragraph - what this page helps you do, written to "you"]

## Sections...

[Body content]

## Next steps

[2-4 links to logical next pages]
```

> **Note:** Do NOT include an `# H1` title in the markdown body. The template renders the frontmatter `title` and `description` as the page header automatically. Starting your markdown with `# Title` creates a duplicate heading.

### Frontmatter requirements

All docs pages must include at minimum:

```elixir
%{
  title: "Page Title",
  description: "One sentence - appears as subtitle and in search/SEO.",
  category: :docs,
  order: 10,
  tags: [:docs, :section_tag],
  draft: false  # or true for unpublished pages
}
```

Optional fields: `audience`, `learning_outcomes`, `prerequisites`, `related`, `legacy_paths`, `menu_label`.

### No bot signatures

Do **not** include `Generated by Jido Documentation Writer Bot` or any run ID footers. Pages are authored content.

## Code examples

### Fenced code blocks

Use standard markdown fenced code blocks with the language identifier:

````markdown
```elixir
defmodule MyApp.CounterAgent do
  use Jido.Agent,
    name: "counter_agent",
    schema: [
      count: [type: :integer, default: 0]
    ]
end
```
````

### Code block conventions

1. **Always specify the language** - `elixir`, `shell`, `json`, etc.
2. **Use realistic module names** - `MyApp.CounterAgent`, not `Foo` or `MyModule`.
3. **Use realistic data** - `%User{name: "Jane", email: "jane@example.com"}`, not `data` or `some_value`.
4. **Show complete, compilable snippets** - no `...` or `# rest of code`. If a snippet is partial, frame it clearly: "Add this to your existing module."
5. **Shell commands use `shell`**, not `bash` or `sh`:

````markdown
```shell
mix deps.get
mix compile
```
````

6. **IEx sessions use `elixir`** with `iex>` prompts and `#=>` for return values:

```elixir
iex> agent = MyApp.CounterAgent.new()
iex> agent.state.count
#=> 0
```

7. **Keep blocks short** - Under 25 lines. If longer, split into multiple blocks with prose between them explaining each step.
8. **No comments that restate the code** - Only comment to explain *why*, never *what*.

### Before/after pattern

When showing improvement, use the before/after pattern from the manifesto:

```elixir
# Before: manual, error-prone
case validate(data) do
  {:ok, valid} -> ...
end

# After: with Jido
agent |> cmd({ValidateAction, %{data: data}})
```

### Dependencies in examples

All examples use Hex dependencies, never workspace/git references:

```elixir
defp deps do
  [
    {:jido, "~> 2.0"},
    {:jido_ai, "~> 0.1"}
  ]
end
```

## Standard page blocks

### Next Steps (required on every page)

Every page ends with a "Next Steps" section containing 2–4 links. Use a brief intro sentence, then a bullet list:

```markdown
## Next Steps

Now that you've defined your first agent, continue building.

- [Build Your First LLM Agent](/docs/learn/first-llm-agent) - add AI capabilities
- [Actions and Validation](/docs/learn/actions-validation) - deeper dive into action schemas
- [Agents concept](/docs/concepts/agents) - understand the data-first model
```

**Rules:**
- Links use relative site paths (`/docs/learn/...`), never full URLs
- Each link has a dash-separated description after the link text
- Order by logical next step, not alphabetically
- First link should be the most common "what's next"

### Package links

When referencing Jido ecosystem packages, link to `/ecosystem/<package>` for package overviews and HexDocs for API details:

```markdown
See the [jido ecosystem page](/ecosystem/jido) for an overview, or dive into the [API docs](https://hexdocs.pm/jido).
```

Never link to `/docs/reference/packages/` - that section has been removed.

### Callout blocks

Use blockquotes with bold labels for warnings, tips, and notes:

```markdown
> **Note:** Agents are immutable structs, not processes. See [Agent Runtime](/docs/concepts/agent-runtime) for the process layer.

> **Warning:** Never store API keys in `config/config.exs`. Use `config/runtime.exs` for all secrets.

> **Tip:** Run `cmd/2` in IEx to inspect state transitions interactively.
```

## Page footer

The docs page footer has two layers, both rendered by the template:

### Content-area footer (inside max-width container)

Sits within the content column, constrained to the same max-width as the prose. Contains:

- **Edit this page →** (links to GitHub edit URL for the current page)

### Right sidebar feedback

The "Was this page helpful?" feedback widget renders in the right sidebar below Quick Links (visible on xl+ screens). It is part of the `docs_right_sidebar` component.

### Marketing footer (full-width, site-wide)

The same `marketing_footer` component used across the rest of the site renders below the content-area footer. It spans the full width of the docs main area and includes:

- Logo + tagline + status indicator
- PROJECT / RESOURCES / SOCIAL / PACKAGES nav columns
- Bottom bar with Apache License 2.0 + Jido version

This ensures every page on the site - docs, marketing, ecosystem - has a consistent bottom-of-page experience. Copyright and license info live only in the marketing footer, not duplicated in the content footer.

## Heading conventions

- **H1** (`#`) - Do not use in markdown body. The template renders the frontmatter `title` as the page heading.
- **H2** (`##`) - Major sections (these appear in the right-side TOC)
- **H3** (`###`) - Subsections within an H2
- **H4+** - Avoid; flatten structure instead

Keep heading text short, under 6 words. Use sentence case, not title case:

```markdown
## Configure runtime secrets     ← good
## Configuration Of Runtime Secrets  ← bad
```

## Link conventions

| Destination | Format |
|-------------|--------|
| Another docs page | `[Page title](/docs/section/slug)` |
| Ecosystem package | `[jido](/ecosystem/jido)` |
| HexDocs API | `[Jido.Agent](https://hexdocs.pm/jido/Jido.Agent.html)` |
| GitHub repo | `[GitHub](https://github.com/agentjido/jido)` |
| Example page | `[Counter Agent example](/examples/counter-agent)` |

Never use bare URLs in prose. Always use descriptive link text - never "click here" or "this page".

## Prose style

- Write to "you", never "users" or "developers" in third person
- Present tense: "This returns a tuple" not "This will return a tuple"
- Active voice: "cmd/2 merges the state" not "The state is merged by cmd/2"
- Max 3 sentences per paragraph
- Cut hedge words: "basically", "simply", "just", "actually", "quite"
- Be specific: "Returns `{agent, directives}`" not "Returns a result"
- Don't explain Elixir basics to Elixir developers
- **No em dashes.** Use a regular hyphen ` - ` for parenthetical asides and link descriptions. Restructure sentences to avoid dashes when possible.
- **No emoji.** Keep prose and headings free of emoji characters.
- Follow the full voice, tone, and terminology conventions in `specs/style-voice.md`.
